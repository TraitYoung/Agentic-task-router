## 三、请求链路（从浏览器到图）

1. **浏览器** → `POST /api/chat/stream`（Next）或直连 `POST http://127.0.0.1:8000/api/v1/chat/stream`（FastAPI）。  
2. **Next**（`frontend/app/api/chat/stream/route.ts`）将 `x-session-id`、`x-trace-id` 转发到 FastAPI。  
3. **`main.chat_stream_api`** / **`chat_api`**（`ChatRequest` 含 `workflow_mode`）：  
   - **`default`**：`get_default_router_runner()`（默认 `run_default_router_turn`）→ Redis 热上下文 + `run_router_traced(router_graph, …)`；可通过 **`AX_DEFAULT_ROUTER_BACKEND`** 换实现（兼容 `AX_CABINET_BACKEND`）。  
   - **`dev_pipeline`**：调用 **`orchestrator.run_dev_pipeline`**（阶段1配置化 + 阶段2并行实现/测试 + merge 汇总；步骤间仍采用 JSON 摘要预算）；不经过主图 **parser**；`intent` 为合成的 `TaskIntent`。  
   - 拼 `reply` 前缀（`dev`）、写回 Redis；SSE meta 含 **`workflow_mode`**。  
4. **`agents/cabinet/graph.py`**（仅 `workflow_mode=default`）：parser → `route_by_intent` → 某一助理节点 → `END`（`agents/router.py` 仅兼容 re-export）。

## 四、HTTP API 摘要（FastAPI）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/health` | 探活；顺带尝试 Redis `ping`。 |
| POST | `/api/v1/chat` | 请求体：`text`、可选 **`workflow_mode`**：`default` \| `dev_pipeline`。响应：`session_id`、`reply`、`intent`、`trace_id`、`trace`。 |
| POST | `/api/v1/chat/stream` | 同上；SSE meta 额外含 **`workflow_mode`**。 |
| GET | `/api/v1/chat/history` | 需 `x-session-id`；Redis 最近轮次。 |
| POST | `/api/v1/chat/export` | 导出 `output/chats/*.jsonl`。 |

**追踪相关**：不传 `x-trace-id` 时服务端生成 UUID；`trace` 中每一步含 `node`、`duration_ms`、`keys_written`、`summary`（意图与回复在 summary 中截断，避免过大 payload）。`__router__` 为合成步骤，与 `route_by_intent` 的 `route_key` / `next_node` 对齐。

## 五、默认路由内核（次能力，`workflow_mode=default`）

### 5.1 状态 `GraphState`（TypedDict）

- `current_input`、`thread_id`、`recent_history`：由 API 注入；parser 消费历史拼进 Prompt。  
- `intent`：`TaskIntent`（parser 产出）。  
- `final_response`、`active_task_type`：各助理节点产出；API 层用 `active_task_type` 映射回复前缀。

### 5.2 节点与边（与代码一致，偏教学/测试）

- **入口**：`parser`（`node_parser`）：`with_structured_output(TaskIntent)`；Q1/Q2 时 `PersonaMemory.save_memory`。  
- **条件边**：`route_by_intent` →  
  `emotion_route` | `jean_route` | `bit_route` | `juzheng_route`  
- **终点节点**：`emotion_agent`、`jean_agent`、`bit_agent`、`juzheng_agent` → `END`。

**注意**：高痛感（`pain_level > 6`）走 **`emotion_route`**，医疗红线话术在 emotion Prompt 中处理，**没有单独的 `medical_agent` 节点**。战略对应 **`juzheng_agent`**，技术/工具链对应 **`bit_agent`**（内含 `create_react_agent` + `BIT_TOOLS`），文档/RAG 材料对应 **`jean_agent`**。

### 5.3 `TaskIntent`（`schemas/protocols.py`）

- `task_type`：`emotion` | `jean` | `bit` | `juzheng` | `unknown`  
- `urgency_level`：1–5；`pain_level`：1–10（>6 强制情绪路径）。  
- `quadrant`：Q1–Q4（艾森豪威尔）；影响是否写入 L3。

### 5.4 Bit 特殊路径

- 关键词命中「SFT / jsonl / 日志清洗」等时，可在节点内优先 `execute_python` 调 `tools/logs_to_sft.py` 流水线。  
- 另有一条基于 **用户原文** 的清洗关键词短路，在 `route_by_intent` 中可强制 `bit_route`（与意图模型并行存在，读代码时注意优先级）。

## 六、记忆层（`memory/database.py`）

- **主表** `memory_matrix`：`thread_id`、`quadrant`、`content`、`status`、`created_at` 等。  
- **FTS5** `memory_fts`：**external-content** + **触发器**，与主表同步。  
- **向量表** `memory_embeddings`：Q2 等内容的 float32 BLOB（维度与 `get_embedding` 一致，常见 1536）。  
- **写入时机**：parser 内 Q1/Q2；**向量**通常由 `scripts/migration.py` 批处理补写，避免阻塞在线写入路径。

---

## 七、Hybrid RAG（`hybrid_engine.py` + `tools/ai_client.py`）

- **关键词路**：FTS5 + BM25（`bm25(memory_fts)`），可按 `thread_id`、`quadrant` 过滤。  
- **向量路**：BLOB → numpy，与 query embedding 余弦相似度。  
- **融合**：RRF（Reciprocal Rank Fusion），对两路排名加权求和，不强行统一分数尺度。  
- **Jean 节点**：`HybridRetriever.search_hybrid`；本会话无命中时可放宽 `thread_id` 再查全库 Q2（代码内已写兜底）。

---

## 八、前端（`frontend/`）

- **聊天**：SSE 解析首包 meta → 更新路由展示与 **全链路追踪** 列表；`delta` 拼接打字机效果。  
- **清洗模式**：通过合成 prompt 触发 Bit 侧流水线或本地工具链（与 `page.tsx` 逻辑一致）。  
- **环境**：后端地址由 `frontend/lib/backend.ts`（或环境变量）决定，需与 uvicorn 端口一致。

## 十一、设计取舍（面试 / 评审可讲）

1. **结构化意图**：用 Pydantic + Structured Output 替代手写解析，降低标签漂移与路由错误。  
2. **痛感优先**：>6 一律走情绪路径，安全优先于 `task_type`。  
3. **Q1/Q2 才落 L3**：Q2 面向长期战略回忆并向量化；控制成本与噪声。  
4. **RRF 而非线性加权分数**：BM25 与余弦尺度不同，RRF 更稳。  
5. **FTS5 external-content**：单写主表、触发器维护索引，避免双写不一致。  
6. **追踪用 `stream(updates)`**：不改各节点签名即可拿到每步 state 增量；`__router__` 显式还原条件边决策。

## 十三、常见追问（简答）

- **LangGraph 与 LangChain**：LangGraph 负责有分支、有状态的多步图；LangChain 提供模型、消息、预置 Agent 抽象。  
- **为何 SSE 仍是「先整段生成再切分」**：当前图为同步 `invoke`/`stream` 聚合结果，再按块推送，实现简单；真 token 级流式需改节点内 `stream` 与协议。  
- **多用户**：区分 `thread_id`（及可选 `user_id`）即可；记忆与检索均已按 `thread_id` 过滤或可配置全库兜底。  
- **向量库迁移**：保持「主键 + float32 向量」语义，可迁到 pgvector / 专用向量库，RRF 层仍可复用排名思想。
